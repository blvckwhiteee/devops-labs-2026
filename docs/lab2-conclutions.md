# Висновки до лабораторної роботи №2

У межах лабораторної роботи були виконані дослідження для Python-застосунку, Golang-застосунку та практична контейнеризація сервісів із попередньої лабораторної роботи. Основна мета полягала  в порівнянні різних підходів упаковки застосунків, оцінці їхніх сильних і слабких сторін та перевірці практичної придатності цих підходів.

## Що було зроблено

Для Python-проєкту було реалізовано три варіанти контейнеризації:
- неоптимізований `Dockerfile`, у якому код копіюється до встановлення залежностей
- оптимізований `Dockerfile`, у якому залежності встановлюються окремим стабільним шаром
- Alpine-варіант образу для порівняння з Debian-based образом

Після цього були виконані повторні збірки після малої зміни у вихідному коді, а також проведений окремий експеримент після додавання залежності `numpy` і нового endpoint для множення матриць.

Для Golang-проєкту було реалізовано:
- single-stage образ
- multi-stage образ із `scratch`
- multi-stage образ із `distroless`

Для кожного варіанта були зібрані метрики часу збірки, розміру образу та перевірена працездатність контейнера.

У практичній частині було контейнеризовано стек із ЛР1:
- `web application`
- `nginx`
- `database`

Також був доданий окремий сервіс міграції, налаштована окрема мережа, persistent volume для БД та залежність старту сервісів від `healthcheck`.

## Отримані результати

### Python-застосунок

Найважливішим результатом у Python-частині стало порівняння неоптимізованого й оптимізованого порядку шарів у `Dockerfile`.

Для `Dockerfile.bad`:
- initial build: близько `65.42 s`
- rebuild після малої зміни: близько `18.91 s`
- шар `pip install` не кешувався

Для `Dockerfile.good`:
- initial build: близько `18.17 s`
- rebuild після малої зміни: близько `1.22 s`
- шар `pip install` залишався в кеші

Це показало, що головний виграш від правильного шарування полягає не у зменшенні розміру образу, а у різкому прискоренні повторних збірок.

Після додавання `numpy` і endpoint для множення матриць було отримано такі результати:
- Debian-based image: близько `398.62 MiB`, build time близько `22.64 s`
- Alpine-based image: близько `52.13 MiB`, build time близько `31.01 s`

При цьому Debian збирався швидше, а Alpine залишався значно компактнішим.

### Musl vs glibc

Окремий DNS-експеримент показав, що однакові DNS-параметри Docker не гарантують однакову поведінку контейнерів на базі Ubuntu і Alpine. У сценарії з `search domain` Ubuntu-контейнер успішно резолвив повне ім’я сервісу, а Alpine-контейнер поводився інакше.

Це показало, що різниця між `glibc` і `musl` впливає не тільки на сумісність пакетів, а й на runtime-поведінку системних компонентів, зокрема DNS resolver-а.

### Golang-застосунок

Для Golang-проєкту були отримані такі результати:
- single-stage: близько `496.07 MiB`, build time близько `60.67 s`
- `scratch`: близько `4.54 MiB`, build time близько `29.79 s`
- `distroless`: близько `5.21 MiB`, build time близько `34.90 s`

Single-stage варіант виявився найпростішим для побудови, але містив багато зайвого: Go toolchain, вихідний код, модулі та інші build-артефакти. Варіанти `scratch` і `distroless` дали значно менші runtime-образи.

Водночас з’ясувалося, що для цього проєкту недостатньо копіювати лише бінарник: застосунок також потребує каталог `templates/`, який має бути явно включений у фінальний образ.

### Практична частина

У результаті practical part було реалізовано контейнеризований стек:
- `db` на базі MariaDB
- `migrate` як окремий одноразовий сервіс
- `web` як окремий контейнерний сервіс
- `nginx` як reverse proxy


## З якими труднощами довелось зіткнутися

У Python-експерименті довелось окремо перевіряти коректність замірів, тому що прогрітий кеш Docker міг давати дуже малий `initial build time`, який не слід було сприймати як повністю “холодну” збірку. Для цього в пайплайнах було явно розділено сценарії з кешем і без кешу.

Ще однією важливою особливістю стало те, що Alpine не виявився “проблемним” для `numpy` у жорсткому вигляді, як це часто траплялося раніше. Об’єктивна причина полягала в тому, що в поточному середовищі використовувалися готові `musllinux` wheels, тому пакет не вимагав повної компіляції з сирців.

У DNS-експерименті складність полягала в тому, що різницю між Ubuntu та Alpine неможливо було пояснити лише фінальним output команди `getent hosts`. Довелось аналізувати логи `dnsmasq`, щоб побачити, які саме DNS-запити реально формувалися в кожному контейнері.

У Golang-частині важливою практичною проблемою було те, що мінімальний образ із `scratch` не можна було зробити лише з одного бінарника: застосунок читав шаблони з файлової системи, отже ці runtime-файли теж потрібно було переносити у фінальний образ.

У practical part найбільше проблем виникло на інтеграційному рівні. Спочатку сервіс міграції запускався некоректно через жорсткий `ENTRYPOINT` у Dockerfile, який заважав правильно перевизначати поведінку контейнера в Compose.


## Висновки

Головний висновок для Python-проєктів полягає в тому, що порядок шарів у Dockerfile критично впливає на вартість повторної збірки. Якщо код копіюється до встановлення залежностей, то навіть мінімальна зміна змушує Docker перевиконувати зайві кроки. Оптимізований Dockerfile слід будувати так, щоб залежності потрапляли в окремий стабільний шар.

Другий висновок полягає в тому, що менший базовий образ не гарантує кращий результат по всіх метриках. Alpine може значно зменшити розмір фінального образу, але не обов’язково пришвидшить збірку. Для проєктів із нативними залежностями потрібна реальна перевірка на цільових версіях пакетів, а не загальні припущення.

Третій висновок стосується того, що вибір базового образу впливає не тільки на розмір і сумісність бібліотек, а й на системну поведінку контейнера. Експеримент із DNS показав, що `glibc` і `musl` можуть давати різний результат навіть за однакових Docker DNS-параметрів.

Для Golang-застосунків multi-stage build є найбільш доцільним підходом. Single-stage образи містять занадто багато зайвих компонентів, тоді як `scratch` і `distroless` дозволяють отримати дуже компактні runtime-образи. При цьому потрібно уважно перевіряти, чи не залежить застосунок від додаткових runtime-файлів.

Для багатосервісних систем контейнеризація не зводиться лише до написання `docker-compose.yml`. Потрібно продумувати порядок запуску, готовність сервісів, ізоляцію мережі, окремий життєвий цикл міграцій і збереження даних. Без цього стек може формально запускатися, але не бути надійним у реальному використанні.

## Рекомендації щодо упаковки застосунків у контейнери

- Для Python-застосунків відокремлювати шар залежностей від шару прикладного коду.
- Не вибирати Alpine автоматично лише через менший розмір; оцінювати і час збірки, і сумісність залежностей.
- Для Go за замовчуванням використовувати multi-stage build.
- Для production-використання Go-застосунків розглядати `distroless` як хороший компроміс між мінімальним розміром і практичністю.
- Для контейнеризації багатосервісних систем окремо планувати healthcheck, мережу, volume і порядок старту сервісів.
- У випадках, де сервіс уже має окремий інструмент міграції БД, не змішувати його із запуском веб-сервера в одному процесі, а виносити в окремий етап або сервіс.
